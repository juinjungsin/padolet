Step 0 — 기획: PRD 작성
무엇을 만들 것인지 정의하는 단계
0-1
요구사항 정리
어떤 제품을 만들지 한 문단으로 적는다. padolet의 경우: "강사와 수강생이 QR코드로 입장해서, 포스트잇을 붙이고 실시간 채팅하는 교육용 보드"
0-2
핵심 기능 목록 작성
• QR + 6자리 코드 입장
• 포스트잇 5열 그리드 (실시간)
• 대화창 (우측 1/4)
• 파일/스크린샷 공유 (50MB, 20개)
• Admin 레포트 (MD 다운로드)
0-3
기술스택 결정
DB를 Firestore로 선택한 이유: 실시간 동기화가 내장(onSnapshot)되어 있어 별도 WebSocket 서버가 불필요. 교육 세션 50명 이하 규모에서 무료 티어로 충분.
0-4
PRD 문서 작성
사용자 흐름, 화면 구성, 데이터 모델, 비기능 요구사항, 빌딩 계획을 하나의 PRD.md 파일로 정리. 이 문서가 전체 빌딩의 설계도 역할.

왜 GitHub Pages가 아닌 Vercel인가
GitHub Pages는 정적 파일(HTML/CSS/JS)만 호스팅한다. padolet은 Next.js의 API Routes(서버 기능), NextAuth(Google OAuth 콜백), SSR(동적 라우팅)을 사용하므로 서버 실행이 가능한 Vercel이 필요하다. GitHub은 소스 코드 저장소, Vercel은 빌드+실행+호스팅 담당.

빌딩 중 만난 문제와 해결
실제 빌딩 과정에서 발생한 에러들
에러 1 — Firebase auth/invalid-api-key
원인: 환경변수(.env.local) 없이 빌드하면 Firebase SDK가 빈 API key로 초기화 시도.
해결: firebase.ts를 lazy init 방식으로 변경. config가 없으면 초기화를 지연시켜 빌드 에러 방지.
에러 2 — Firestore 복합 인덱스 필요
원인: createdBy + createdAt 복합 쿼리에 인덱스가 없음.
해결: 에러 메시지에 포함된 Firebase Console 링크를 열어 "인덱스 만들기" 클릭. 1~2분 대기.
에러 3 — git commit Author identity unknown
원인: 새 폴더에서 git init 후 user.name/email 미설정.
해결: git config user.name "이름" + git config user.email "이메일"
에러 4 — 400 redirect_uri_mismatch
원인: Google OAuth 클라이언트에 프로덕션 도메인의 리디렉션 URI가 등록되지 않음.
해결: Google Cloud Console에서 승인된 리디렉션 URI에 프로덕션 URL 추가.

초보자를 위한 요약
이 프로젝트의 핵심은 "Google 생태계 통합"이다. Firestore(DB) + Firebase Storage(파일) + Google OAuth(인증)를 하나의 Firebase 프로젝트로 묶고, Next.js(프론트+백엔드)를 Vercel에 배포하면 서버 관리 없이 풀스택 웹앱이 동작한다. 전체 인프라 비용: 무료 티어 범위 내.
